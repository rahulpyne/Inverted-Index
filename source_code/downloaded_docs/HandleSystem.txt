   <h1 class="firstHeading" id="firstHeading" lang="en">
    Handle System
   </h1>
   <div class="mw-body-content" id="bodyContent">
    <div id="siteSub">
     From Wikipedia, the free encyclopedia
    </div>
    <div id="contentSub">
    </div>
    <div class="mw-jump" id="jump-to-nav">
     Jump to:
     <a href="#mw-head">
      navigation
     </a>
     ,
     <a href="#p-search">
      search
     </a>
    </div>
    <div class="mw-content-ltr" dir="ltr" id="mw-content-text" lang="en">
     <p>
      The
      <b>
       Handle System
      </b>
      is a technology specification for assigning, managing, and resolving
      <a href="/wiki/Persistent_identifier" title="Persistent identifier">
       persistent identifiers
      </a>
      for
      <a class="mw-redirect" href="/wiki/Digital_object" title="Digital object">
       digital objects
      </a>
      and other resources on the Internet. The protocols specified enable a distributed computer system to store identifiers (names, or handles) of digital resources and resolve those handles into the information necessary to locate, access, and otherwise make use of the resources. That information can be changed as needed to reflect the current state and/or location of the identified resource without changing the handle.
     </p>
     <p>
      The Handle System was developed by
      <a href="/wiki/Bob_Kahn" title="Bob Kahn">
       Bob Kahn
      </a>
      , co-inventor of the
      <a class="mw-redirect" href="/wiki/Internet_Protocol_Suite" title="Internet Protocol Suite">
       TCP/IP
      </a>
      protocols that underlie the operation of the
      <a href="/wiki/Internet" title="Internet">
       Internet
      </a>
      , with support from the Defense Advanced Research Projects Agency
      <a href="/wiki/DARPA" title="DARPA">
       DARPA
      </a>
      at the
      <a href="/wiki/Corporation_for_National_Research_Initiatives" title="Corporation for National Research Initiatives">
       Corporation for National Research Initiatives
      </a>
      (CNRI), which continues to develop and manage it. The Handle System is currently in use in several applications.
      <sup class="reference" id="cite_ref-1">
       <a href="#cite_note-1">
        [1]
       </a>
      </sup>
     </p>
     <p>
      The Handle System enables management of objects as
      <a class="mw-redirect" href="/wiki/First-class_object" title="First-class object">
       first class entities
      </a>
      , rather than as packets of bits with dependency on other attributes such as locations. It emerged as part of a wider Framework for Distributed Digital Object Services
      <sup class="reference" id="cite_ref-2">
       <a href="#cite_note-2">
        [2]
       </a>
      </sup>
      but has been used in independent applications. The system is designed to be scalable
      <sup class="reference" id="cite_ref-3">
       <a href="#cite_note-3">
        [3]
       </a>
      </sup>
      to very large numbers of entities without performance degradation, to allow distributed administration, and to enable resolution to multiple pieces of current data (each of which may be separately managed). It also has further optional features such as
      <a href="/wiki/Public_key_infrastructure" title="Public key infrastructure">
       public key infrastructure
      </a>
      capability to enable trust applications.
     </p>
     <p>
      Resolution is the process in which an identifier is the input request to a network service to receive in return a specific output of one or more pieces of current information (state data) related to the identified entity: e.g., a location (URL). The
      <a href="/wiki/Domain_Name_System" title="Domain Name System">
       Domain Name System
      </a>
      resolves domain names meaningful to humans into numerical
      <a class="mw-redirect" href="/wiki/IP_addresses" title="IP addresses">
       IP addresses
      </a>
      (locations of file servers). The Handle System is compatible with DNS but does not necessarily require it, unlike persistent identifiers such as
      <a class="mw-redirect" href="/wiki/Persistent_Uniform_Resource_Locator" title="Persistent Uniform Resource Locator">
       PURLs
      </a>
      or
      <a href="/wiki/Archival_Resource_Key" title="Archival Resource Key">
       ARKs
      </a>
      which utilise domain names and are therefore ultimately constrained by them. Other significant differences include the administrative
      <a href="/wiki/Granularity" title="Granularity">
       granularity
      </a>
      possible with the Handle System (administrators can be different for each handle, and there can also be more than one per handle) and the option for extensible multiple
      <a href="/wiki/Data_type" title="Data type">
       data types
      </a>
      to be assigned.
      <sup class="reference" id="cite_ref-4">
       <a href="#cite_note-4">
        [4]
       </a>
      </sup>
     </p>
     <p>
      DNS has well-recognised problems of security and updating which suggest that it will not be sufficient to assume that existing DNS technology can simply be adapted to deal with new requirements. By explicitly separating names from all associated data, including location, the Handle System addresses a key requirement of future internet architecture. A joint research project by the
      <a class="mw-redirect" href="/wiki/MIT_Laboratory_for_Computer_Science" title="MIT Laboratory for Computer Science">
       MIT Laboratory for Computer Science
      </a>
      and
      <a href="/wiki/Air_Force_Research_Laboratory" title="Air Force Research Laboratory">
       Air Force Research Laboratory
      </a>
      argued that "it is possible to separate the ideas of location and identity, both of which are represented by the IP address in today's Internet, ... the resulting architecture facilitates mobility as well as solving other problems with today's network".
      <sup class="reference" id="cite_ref-ref1_5-0">
       <a href="#cite_note-ref1-5">
        [5]
       </a>
      </sup>
     </p>
     <p>
     </p>

     <p>

     <h2>
      <span class="mw-headline" id="Specifications">
       Specifications
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Handle_System&amp;action=edit&amp;section=1" title="Edit section: Specifications">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      The Handle System is defined in informational
      <a href="/wiki/Request_for_Comments" title="Request for Comments">
       RFCs
      </a>
      3650,
      <sup class="reference" id="cite_ref-6">
       <a href="#cite_note-6">
        [6]
       </a>
      </sup>
      3651
      <sup class="reference" id="cite_ref-7">
       <a href="#cite_note-7">
        [7]
       </a>
      </sup>
      and 3652
      <sup class="reference" id="cite_ref-8">
       <a href="#cite_note-8">
        [8]
       </a>
      </sup>
      of the
      <a href="/wiki/Internet_Engineering_Task_Force" title="Internet Engineering Task Force">
       Internet Engineering Task Force (IETF)
      </a>
      ; it includes an open set of protocols, a namespace, and a reference implementation of the protocols. Handles resolve to
      <a href="/wiki/Data_type" title="Data type">
       typed data
      </a>
      . Documentation, software, and related information is provided by CNRI on a dedicated website
      <sup class="reference" id="cite_ref-9">
       <a href="#cite_note-9">
        [9]
       </a>
      </sup>
      Each handle may have its own administrator(s) and administration of these handles can be done in a distributed environment. The name-to-value bindings may also be secured, both via signatures to verify the data and via challenge response to verify the transmission of the data, allowing handles to be used in trust management applications. The syntax of the handle encompasses any
      <a href="/wiki/Unicode" title="Unicode">
       Unicode
      </a>
      character and leaves the string construction to the assigner (thereby allowing inclusion of existing identifier strings if desired).
     </p>
     <p>
      Implementation of the Handle System consists of Local Handle Services, each of which is made up of one or more sites that provide the servers that store specific handles. The Global Handle Registry is a unique Local Handle Service which stores information on the prefixes (also known as naming authorities) within the Handle System and can be queried to find out where specific handles are stored on other Local Handle Services within this distributed system.
     </p>
     <p>
      Handles can be used natively, or expressed as
      <a href="/wiki/Uniform_Resource_Name" title="Uniform Resource Name">
       Uniform Resource Names (URNs)
      </a>
      or
      <a href="/wiki/Uniform_Resource_Identifier" title="Uniform Resource Identifier">
       Uniform Resource Identifiers (URIs)
      </a>
      . Although the Handle System is not currently a registered stand-alone implementation of URI or URN, it is a part of the info URI
      <sup class="reference" id="cite_ref-10">
       <a href="#cite_note-10">
        [10]
       </a>
      </sup>
      specification,
      <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc4452" rel="nofollow">
       RFC 4452
      </a>
      .
      <sup class="reference" id="cite_ref-11">
       <a href="#cite_note-11">
        [11]
       </a>
      </sup>
      Handles may also be expressed as Uniform Resource Locators (URLs), by the use of a http proxy server.
      <sup class="reference" id="cite_ref-12">
       <a href="#cite_note-12">
        [12]
       </a>
      </sup>
     </p>
     <h2>
      <span class="mw-headline" id="Implementation">
       Implementation
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Handle_System&amp;action=edit&amp;section=2" title="Edit section: Implementation">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      The Handle System website provides a series of implementation tools, notably the HANDLE.NET Software
      <sup class="reference" id="cite_ref-13">
       <a href="#cite_note-13">
        [13]
       </a>
      </sup>
      and HANDLE.NET Client Libraries.
      <sup class="reference" id="cite_ref-14">
       <a href="#cite_note-14">
        [14]
       </a>
      </sup>
      Handle clients can be embedded in end user software (e.g., a web browser) or in server software (e.g., a web server) and extensions are already available for
      <a href="/wiki/Adobe_Acrobat" title="Adobe Acrobat">
       Adobe Acrobat
      </a>
      <sup class="reference" id="cite_ref-15">
       <a href="#cite_note-15">
        [15]
       </a>
      </sup>
      and
      <a href="/wiki/Firefox" title="Firefox">
       Firefox
      </a>
      .
      <sup class="reference" id="cite_ref-16">
       <a href="#cite_note-16">
        [16]
       </a>
      </sup>
     </p>
     <p>
      Handle client software libraries are available in both C and Java. Some applications have developed specific add-on tools, e.g., for the DOI System.
      <sup class="reference" id="cite_ref-17">
       <a href="#cite_note-17">
        [17]
       </a>
      </sup>
     </p>
     <p>
      The interoperable network of distributed handle resolver servers (also known as the Proxy Server System) are linked through a Global Resolver (which is one logical entity though physically decentralised and mirrored). Users of Handle System technology obtain a handle prefix created in the Global Handle Registry.
      <sup class="reference" id="cite_ref-18">
       <a href="#cite_note-18">
        [18]
       </a>
      </sup>
      The Global Handle Registry maintains and resolves the prefixes of locally maintained handle services. Any local handle service can, therefore, resolve any handle through the Global Resolver.
     </p>
     <p>
      Handles (identifiers) are passed by a client, as a query of the naming authority/prefix, to the Handle System's Global Handle Registry (GHR). The GHR responds by sending the client the location information for the relevant Local Handle Service (which may consist of multiple servers in multiple sites); a query is then sent to the relevant server within the Local Handle Service. The Local Handle Service returns the information needed to acquire the resource, e.g., a URL which can then be turned into an HTTP re-direct. (Note: if the client already has information on the appropriate LHS to query, the initial query to GHR is omitted)
     </p>
     <p>
      Though the original model from which the Handle System derives dealt with management of digital objects, the Handle System does not mandate any particular model of relationships between the identified entities, nor is it limited to identifying only digital objects: non-digital entities may be represented as a corresponding digital object for the purposes of digital object management. Some care is needed in the definition of such objects and how they relate to non-digital entities; there are established models that can aid in such definitions (e.g.,
      <a href="/wiki/Functional_Requirements_for_Bibliographic_Records" title="Functional Requirements for Bibliographic Records">
       Functional Requirements for Bibliographic Records (FRBR)
      </a>
      ,
      <a class="mw-redirect" href="/wiki/CIDOC_CRM" title="CIDOC CRM">
       CIDOC CRM
      </a>
      , and
      <a href="/wiki/Indecs_Content_Model" title="Indecs Content Model">
       indecs content model
      </a>
      . Some applications have found it helpful to marry such a framework to the handle application: for example, the Advanced Distributed Learning (ADL) Initiative
      <sup class="reference" id="cite_ref-19">
       <a href="#cite_note-19">
        [19]
       </a>
      </sup>
      brings together Handle System application with existing standards for distributed learning content, using a Shareable Content Object Reference Model (SCORM),
      <sup class="reference" id="cite_ref-20">
       <a href="#cite_note-20">
        [20]
       </a>
      </sup>
      and the
      <a class="mw-redirect" href="/wiki/Digital_Object_Identifier" title="Digital Object Identifier">
       Digital Object Identifier (DOI) system
      </a>
      implementation of the Handle System has adopted it together with the
      <a href="/wiki/Indecs_Content_Model" title="Indecs Content Model">
       indecs
      </a>
      framework to deal with
      <a href="/wiki/Semantic_interoperability" title="Semantic interoperability">
       semantic interoperability
      </a>
      .
     </p>
     <p>
      The Handle System also makes explicit the importance of organizational commitment to a persistent identifier scheme, but does not mandate one model for ensuring such commitment. Individual applications may choose to establish their own sets of rules and social infrastructure to ensure persistence (e.g., when used in the
      <a href="/wiki/DSpace" title="DSpace">
       DSpace
      </a>
      application, and the DOI application).
      <sup class="reference" id="cite_ref-21">
       <a href="#cite_note-21">
        [21]
       </a>
      </sup>
     </p>
     <h2>
      <span class="mw-headline" id="Design_principles">
       Design principles
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Handle_System&amp;action=edit&amp;section=3" title="Edit section: Design principles">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      The Handle system is designed to meet the following requirements to contribute to persistence
      <sup class="reference" id="cite_ref-ref2_22-0">
       <a href="#cite_note-ref2-22">
        [22]
       </a>
      </sup>
      <sup class="reference" id="cite_ref-ref3_23-0">
       <a href="#cite_note-ref3-23">
        [23]
       </a>
      </sup>
     </p>
     <p>
      The identifier string:
     </p>
     <ul>
      <li>
       is not based on any changeable attributes of the entity (location, ownership, or any other attribute that may change without changing the referent’s identity);
      </li>
      <li>
       is opaque (preferably a ‘dumb number’: a well known pattern invites assumptions that may be misleading, and meaningful semantics may not translate across languages and may cause trademark conflicts);
      </li>
      <li>
       is unique within the system (to avoid collisions and referential uncertainty);
      </li>
      <li>
       has optional, but nice to have, features that should be supported (human-readable,cut-and-paste-able, embeddable; fits common systems, e.g., URI specification).
      </li>
     </ul>
     <p>
      The identifier resolution mechanism:
     </p>
     <ul>
      <li>
       is reliable (using redundancy, no single points of failure, and fast enough to not appear broken);
      </li>
      <li>
       is scalable (higher loads simply managed with more computers);
      </li>
      <li>
       is flexible (can adapt to changing computing environments; useful to new applications):
      </li>
      <li>
       is trusted (both resolution and administration have technical trust methods; an operating organization is committed to the long term);
      </li>
      <li>
       builds on open architecture (encouraging the leverage efforts of a community in building applications on the infrastructure);
      </li>
      <li>
       is transparent (users need not know the infrastructure details).
      </li>
     </ul>
     <h2>
      <span class="mw-headline" id="Applications">
       Applications
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Handle_System&amp;action=edit&amp;section=4" title="Edit section: Applications">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      Among the objects that are currently identified by handles are journal articles, technical reports, books, theses and dissertations, government documents, metadata, distributed learning content, and data sets. Handles are being used in digital watermarking applications, GRID applications, repositories, and more. Although individual users may download and use the HANDLE.NET software independently, many users have found it beneficial to collaborate in developing applications in a federation, using common policy or additional technology to provide shared services. As one of the first persistent identifier schemes, the Handle System has been widely adopted by public and private institutions and proven over several years. (See Paradigm, Persistent identifiers.)
      <sup class="reference" id="cite_ref-24">
       <a href="#cite_note-24">
        [24]
       </a>
      </sup>
     </p>
     <p>
      Handle System applications may use handles as simple persistent identifiers (as most commonly used, to resolve to the current URL of an object), or may choose to take advantage of other features. Its support for the simultaneous return as output of multiple pieces of current information related to the object, in defined data structures, enables priorities to be established for the order in which the multiple resolutions will be used. Handles can, therefore, resolve to different digital versions of the same content, to mirror sites, or to different business models (pay vs. free, secure vs. open, public vs. private). They can also resolve to different digital versions of differing content, such as a mix of objects required for a distance-learning course.
     </p>
     <p>
      There are thousands of handle services running today, located in 71 countries, on 6 continents; over 1000 of them run at universities and libraries. Handle services are being run by user federations, national laboratories, universities, computing centers, libraries (national and local), government agencies, contractors, corporations, and research groups. Major publishers use the Handle System for persistent identification of commercially traded and Open Access content through its implementation with the
      <a class="mw-redirect" href="/wiki/Digital_Object_Identifier" title="Digital Object Identifier">
       Digital Object Identifier (DOI) system
      </a>
      .
     </p>
     <p>
      The number of prefixes, which allow users to assign handles, is growing and stands at over 12,000 as of early 2014. There are six top-level Global Handle Registry servers that receive (on average) 68 million resolution requests per month. Proxy servers known to CNRI, passing requests to the system on the Web, receive (on average) 200 million resolution requests per month. (Statistics from Handle Quick Facts.)
      <sup class="reference" id="cite_ref-25">
       <a href="#cite_note-25">
        [25]
       </a>
      </sup>
     </p>
     <p>
      CNRI and
      <a class="mw-redirect" href="/wiki/ITU" title="ITU">
       ITU
      </a>
      (International Telecommunication Union) recently entered into an agreement to collaborate on use of the Handle System (and the Digital Object Architecture more generally) and are working on the specific details of that collaboration; in April 2009 ITU listed the Handle System as an "emerging trend".
      <sup class="reference" id="cite_ref-26">
       <a href="#cite_note-26">
        [26]
       </a>
      </sup>
     </p>
     <h2>
      <span class="mw-headline" id="Licences_and_use_policy">
       Licences and use policy
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Handle_System&amp;action=edit&amp;section=5" title="Edit section: Licences and use policy">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      Handle System, HANDLE.NET and Global Handle Registry are trademarks of the
      <a href="/wiki/Corporation_for_National_Research_Initiatives" title="Corporation for National Research Initiatives">
       Corporation for National Research Initiatives
      </a>
      (CNRI), a non-profit research and development corporation in the USA. The Handle System is the subject of patents by CNRI, which licenses its Handle System technology through a public license,
      <sup class="reference" id="cite_ref-27">
       <a href="#cite_note-27">
        [27]
       </a>
      </sup>
      similar to an open source license, in order to enable broader use of the technology. Handle System infrastructure is supported by prefix registration and service fees, with the majority coming from single prefix holders. The largest current single contributor is the
      <a class="mw-redirect" href="/wiki/International_DOI_Foundation" title="International DOI Foundation">
       International DOI Foundation
      </a>
      . The Public License allows commercial and non-commercial use at low cost of both its patented technology and the reference implementation of the software, and allows the software to be freely embedded in other systems and products. A Service Agreement
      <sup class="reference" id="cite_ref-28">
       <a href="#cite_note-28">
        [28]
       </a>
      </sup>
      is also available for users who intend to provide identifier and/or resolution services using the Handle System technology under the Handle System public license.

     <p>
      The Handle System is the first piece of a long-term digital object architecture. In January 2010 CNRI released its general-purpose Digital Object Repository software,
      <sup class="reference" id="cite_ref-29">
       <a href="#cite_note-29">
        [29]
       </a>
      </sup>
      which is the second major component of this architecture. More information
      <sup class="reference" id="cite_ref-30">
       <a href="#cite_note-30">
        [30]
       </a>
      </sup>
      about the release, including protocol specification, source code and ready-to-use system, clients and utilities, is available.
      <sup class="reference" id="cite_ref-31">
       <a href="#cite_note-31">
        [31]
       </a>
      </sup>
      The third and final piece, the Digital Object Registry, will be released shortly.
      <sup class="noprint Inline-Template" style="white-space:nowrap;">
       [
       <i>
        <a href="/wiki/Wikipedia:Manual_of_Style/Dates_and_numbers#Chronological_items" title="Wikipedia:Manual of Style/Dates and numbers">
         <span title="The time period mentioned near this tag is ambiguous. (August 2016)">
          when?
         </span>
        </a>
       </i>
       ]
      </sup>
     </p>
     <p>
      The continued use and evolution of the Handle System is in no way dependent on these other components, but those already using Handles may find them useful in small or large ways and both are, or soon will be, freely available under an open source style license.
     </p>
