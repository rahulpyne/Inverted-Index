     <p>
      <b>
       Wireless sensor networks
      </b>
      (
      <b>
       WSN
      </b>
      ), sometimes called
      <b>
       wireless sensor and actuator networks
      </b>
      (
      <b>
       WSAN
      </b>
      ),
      <sup class="reference" id="cite_ref-1">
       <a href="#cite_note-1">
        [1]
       </a>
      </sup>
      <sup class="reference" id="cite_ref-2">
       <a href="#cite_note-2">
        [2]
       </a>
      </sup>
      are spatially distributed
      <a class="mw-redirect" href="/wiki/Autonomous" title="Autonomous">
       autonomous
      </a>
      <a href="/wiki/Sensor" title="Sensor">
       sensors
      </a>
      to
      <i>
       monitor
      </i>
      physical or environmental conditions, such as
      <a href="/wiki/Temperature" title="Temperature">
       temperature
      </a>
      ,
      <a href="/wiki/Sound" title="Sound">
       sound
      </a>
      ,
      <a href="/wiki/Pressure" title="Pressure">
       pressure
      </a>
      , etc. and to cooperatively pass their data through the network to a main location. The more modern networks are bi-directional, also enabling
      <i>
       control
      </i>
      of sensor activity. The development of wireless sensor networks was motivated by military applications such as battlefield surveillance; today such networks are used in many industrial and consumer applications, such as industrial process monitoring and control, machine health monitoring, and so on.
     </p>
     <p>
      The WSN is built of "nodes" – from a few to several hundreds or even thousands, where each node is connected to one (or sometimes several) sensors. Each such sensor network node has typically several parts: a
      <a href="/wiki/Radio" title="Radio">
       radio
      </a>
      <a href="/wiki/Transceiver" title="Transceiver">
       transceiver
      </a>
      with an internal
      <a href="/wiki/Antenna_(radio)" title="Antenna (radio)">
       antenna
      </a>
      or connection to an external antenna, a
      <a href="/wiki/Microcontroller" title="Microcontroller">
       microcontroller
      </a>
      , an electronic circuit for interfacing with the sensors and an energy source, usually a
      <a href="/wiki/Battery_(electricity)" title="Battery (electricity)">
       battery
      </a>
      or an
      <a href="/wiki/Embedded_system" title="Embedded system">
       embedded
      </a>
      form of
      <a href="/wiki/Energy_harvesting" title="Energy harvesting">
       energy harvesting
      </a>
      . A
      <a href="/wiki/Sensor_node" title="Sensor node">
       sensor node
      </a>
      might vary in size from that of a shoebox down to the size of a grain of dust, although functioning "motes" of genuine microscopic dimensions have yet to be created. The cost of sensor nodes is similarly variable, ranging from a few to hundreds of dollars, depending on the complexity of the individual sensor nodes. Size and cost constraints on sensor nodes result in corresponding constraints on resources such as energy, memory, computational speed and communications bandwidth. The topology of the WSNs can vary from a simple
      <a href="/wiki/Star_network" title="Star network">
       star network
      </a>
      to an advanced
      <a href="/wiki/Mesh_networking" title="Mesh networking">
       multi-hop
      </a>
      <a href="/wiki/Wireless_mesh_network" title="Wireless mesh network">
       wireless mesh network
      </a>
      . The propagation technique between the hops of the network can be
      <a href="/wiki/Routing" title="Routing">
       routing
      </a>
      or
      <a href="/wiki/Flooding_algorithm" title="Flooding algorithm">
       flooding
      </a>
      .
      <sup class="reference" id="cite_ref-3">
       <a href="#cite_note-3">
        [3]
       </a>
      </sup>
      <sup class="reference" id="cite_ref-4">
       <a href="#cite_note-4">
        [4]
       </a>
      </sup>
     </p>
     <p>
      In
      <a href="/wiki/Computer_science" title="Computer science">
       computer science
      </a>
      and
      <a class="mw-redirect" href="/wiki/Telecommunications" title="Telecommunications">
       telecommunications
      </a>
      , wireless sensor networks are an active research area with numerous workshops and conferences arranged each year, for example
      <a href="/wiki/International_Conference_on_Information_Processing_in_Sensor_Networks" title="International Conference on Information Processing in Sensor Networks">
       IPSN
      </a>
      ,
      <a href="/wiki/Conference_on_Embedded_Networked_Sensor_Systems" title="Conference on Embedded Networked Sensor Systems">
       SenSys
      </a>
      , and
      <a href="/wiki/European_Conference_on_Wireless_Sensor_Networks" title="European Conference on Wireless Sensor Networks">
       EWSN
      </a>
      .
     </p>

     <h2>
      <span class="mw-headline" id="Characteristics">
       Characteristics
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Wireless_sensor_network&amp;action=edit&amp;section=17" title="Edit section: Characteristics">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      The main characteristics of a WSN include:
     </p>
     <ul>
      <li>
       Power consumption constraints for nodes using batteries or
       <a href="/wiki/Energy_harvesting" title="Energy harvesting">
        energy harvesting
       </a>
      </li>
      <li>
       Ability to cope with node failures (
       <a href="/wiki/Resilience_(network)" title="Resilience (network)">
        resilience
       </a>
       )
      </li>
      <li>
       Some mobility of nodes (for highly mobile nodes see
       <a class="mw-redirect" href="/wiki/MWSN" title="MWSN">
        MWSNs
       </a>
       )
      </li>
      <li>
       Heterogeneity of nodes
      </li>
      <li>
       <a href="/wiki/Scalability" title="Scalability">
        Scalability
       </a>
       to large scale of deployment
      </li>
      <li>
       Ability to withstand harsh environmental conditions
      </li>
      <li>
       Ease of use
      </li>
      <li>
       <a class="new" href="/w/index.php?title=Cross-layer&amp;action=edit&amp;redlink=1" title="Cross-layer (page does not exist)">
        Cross-layer
       </a>
       design
       <sup class="reference" id="cite_ref-13">
        <a href="#cite_note-13">
         [13]
        </a>
       </sup>
      </li>
     </ul>
     <p>
      Cross-layer is becoming an important studying area for wireless communications. In addition, the traditional layered approach presents three main problems:
     </p>
     <ol>
      <li>
       Traditional layered approach cannot share different information among different layers ， which leads to each layer not having complete information. The traditional layered approach cannot guarantee the optimization of the entire network.
      </li>
      <li>
       The traditional layered approach does not have the ability to adapt to the environmental change.
      </li>
      <li>
       Because of the interference between the different users, access conflicts, fading, and the change of environment in the wireless sensor networks, traditional layered approach for wired networks is not applicable to wireless networks.
      </li>
     </ol>
     <p>
      So the cross-layer can be used to make the optimal modulation to improve the transmission performance, such as
      <a href="/wiki/Bit_rate" title="Bit rate">
       data rate
      </a>
      , energy efficiency, QoS (
      <a class="mw-redirect" href="/wiki/Quality_of_Service" title="Quality of Service">
       Quality of Service
      </a>
      ), etc.. Sensor nodes can be imagined as small computers which are extremely basic in terms of their interfaces and their components. They usually consist of a
      <i>
       processing unit
      </i>
      with limited computational power and limited memory,
      <i>
       sensors
      </i>
      or
      <a href="/wiki/Microelectromechanical_systems" title="Microelectromechanical systems">
       MEMS
      </a>
      (including specific conditioning circuitry), a
      <i>
       communication device
      </i>
      (usually radio transceivers or alternatively
      <a class="mw-redirect" href="/wiki/Smart_dust" title="Smart dust">
       optical
      </a>
      ), and a power source usually in the form of a battery. Other possible inclusions are
      <a href="/wiki/Energy_harvesting" title="Energy harvesting">
       energy harvesting
      </a>
      modules,
      <sup class="reference" id="cite_ref-14">
       <a href="#cite_note-14">
        [14]
       </a>
      </sup>
      secondary
      <a href="/wiki/Application-specific_integrated_circuit" title="Application-specific integrated circuit">
       ASICs
      </a>
      , and possibly secondary communication interface (e.g.
      <a href="/wiki/RS-232" title="RS-232">
       RS-232
      </a>
      or
      <a class="mw-redirect" href="/wiki/Universal_Serial_Bus" title="Universal Serial Bus">
       USB
      </a>
      ).
     </p>
     <p>
      The base stations are one or more components of the WSN with much more computational, energy and communication resources. They act as a gateway between sensor nodes and the end user as they typically forward data from the WSN on to a server. Other special components in
      <a href="/wiki/Routing" title="Routing">
       routing
      </a>
      based networks are routers, designed to compute, calculate and distribute the routing tables.
     </p>
     <h2>
      <span class="mw-headline" id="Platforms">
       Platforms
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Wireless_sensor_network&amp;action=edit&amp;section=18" title="Edit section: Platforms">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <h3>
      <span class="mw-headline" id="Hardware">
       Hardware
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Wireless_sensor_network&amp;action=edit&amp;section=19" title="Edit section: Hardware">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <div class="hatnote" role="note">
      Main article:
      <a href="/wiki/Sensor_node" title="Sensor node">
       sensor node
      </a>
     </div>
     <p>
      One major challenge in a WSN is to produce
      <i>
       low cost
      </i>
      and
      <i>
       tiny
      </i>
      sensor nodes. There are an increasing number of small companies producing WSN hardware and the commercial situation can be compared to home computing in the 1970s. Many of the nodes are still in the research and development stage, particularly their software. Also inherent to sensor network adoption is the use of very low power methods for radio communication and data acquisition.
     </p>
     <p>
      In many applications, a WSN communicates with a
      <a class="mw-redirect" href="/wiki/Local_Area_Network" title="Local Area Network">
       Local Area Network
      </a>
      or
      <a class="mw-redirect" href="/wiki/Wide_Area_Network" title="Wide Area Network">
       Wide Area Network
      </a>
      through a gateway. The Gateway acts as a bridge between the WSN and the other network. This enables data to be stored and processed by devices with more resources, for example, in a remotely located
      <a href="/wiki/Server_(computing)" title="Server (computing)">
       server
      </a>
      .
     </p>
     <h3>
      <span class="mw-headline" id="Software">
       Software
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Wireless_sensor_network&amp;action=edit&amp;section=20" title="Edit section: Software">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Energy is the scarcest resource of WSN nodes, and it determines the lifetime of WSNs. WSNs may be deployed in large numbers in various environments, including remote and hostile regions, where ad hoc communications are a key component. For this reason, algorithms and protocols need to address the following issues:
     </p>
     <ul>
      <li>
       Increased lifespan
      </li>
      <li>
       Robustness and fault tolerance
      </li>
      <li>
       Self-configuration
      </li>
     </ul>
     <p>
      Lifetime maximization: Energy/Power Consumption of the sensing device should be minimized and sensor nodes should be energy efficient since their limited energy resource determines their lifetime. To conserve power, wireless sensor nodes normally power off both the radio transmitter and the radio receiver when not in use.
     </p>
     <h4>
      <span class="mw-headline" id="Operating_systems">
       Operating systems
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Wireless_sensor_network&amp;action=edit&amp;section=21" title="Edit section: Operating systems">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h4>
     <p>
      <a href="/wiki/Operating_system" title="Operating system">
       Operating systems
      </a>
      for wireless sensor network nodes are typically less complex than general-purpose operating systems. They more strongly resemble
      <a href="/wiki/Embedded_system" title="Embedded system">
       embedded systems
      </a>
      , for two reasons. First, wireless sensor networks are typically deployed with a particular application in mind, rather than as a general platform. Second, a need for low costs and low power leads most wireless sensor nodes to have low-power microcontrollers ensuring that mechanisms such as virtual memory are either unnecessary or too expensive to implement.
     </p>
     <p>
      It is therefore possible to use embedded operating systems such as
      <a href="/wiki/ECos" title="ECos">
       eCos
      </a>
      or
      <a class="mw-redirect" href="/wiki/UC/OS" title="UC/OS">
       uC/OS
      </a>
      for sensor networks. However, such operating systems are often designed with real-time properties.
     </p>
     <p>
      <a href="/wiki/TinyOS" title="TinyOS">
       TinyOS
      </a>
      is perhaps the first
      <sup class="reference" id="cite_ref-15">
       <a href="#cite_note-15">
        [15]
       </a>
      </sup>
      operating system specifically designed for wireless sensor networks. TinyOS is based on an
      <a href="/wiki/Event-driven_programming" title="Event-driven programming">
       event-driven programming
      </a>
      model instead of
      <a class="mw-redirect" href="/wiki/Thread_(computer_science)" title="Thread (computer science)">
       multithreading
      </a>
      . TinyOS programs are composed of
      <i>
       event handlers
      </i>
      and
      <i>
       tasks
      </i>
      with run-to-completion semantics. When an external event occurs, such as an incoming data packet or a sensor reading, TinyOS signals the appropriate event handler to handle the event. Event handlers can post tasks that are scheduled by the TinyOS kernel some time later.
     </p>
     <p>
      <a href="/wiki/LiteOS" title="LiteOS">
       LiteOS
      </a>
      is a newly developed OS for wireless sensor networks, which provides UNIX-like abstraction and support for the C programming language.
     </p>
     <p>
      <a href="/wiki/Contiki" title="Contiki">
       Contiki
      </a>
      is an OS which uses a simpler programming style in C while providing advances such as
      <a href="/wiki/6LoWPAN" title="6LoWPAN">
       6LoWPAN
      </a>
      and
      <a href="/wiki/Protothreads" title="Protothreads">
       Protothreads
      </a>
      .
     </p>
     <h3>
      <span class="mw-headline" id="Online_collaborative_sensor_data_management_platforms">
       Online collaborative sensor data management platforms
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Wireless_sensor_network&amp;action=edit&amp;section=22" title="Edit section: Online collaborative sensor data management platforms">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      Online collaborative sensor data management platforms are on-line database services that allow sensor owners to register and connect their devices to feed data into an online database for storage and also allow developers to connect to the database and build their own applications based on that data. Examples include
      <a href="/wiki/Xively" title="Xively">
       Xively
      </a>
      and the
      <a class="external text" href="http://wikisensing.org" rel="nofollow">
       Wikisensing platform
      </a>
      . Such platforms simplify online collaboration between users over diverse data sets ranging from energy and environment data to that collected from transport services. Other services include allowing developers to embed real-time graphs &amp; widgets in websites; analyse and process historical data pulled from the data feeds; send real-time alerts from any datastream to control scripts, devices and environments.
     </p>
     <p>
      The architecture of the Wikisensing system
      <sup class="reference" id="cite_ref-16">
       <a href="#cite_note-16">
        [16]
       </a>
      </sup>
      describes the key components of such systems to include APIs and interfaces for online collaborators, a middleware containing the business logic needed for the sensor data management and processing and a storage model suitable for the efficient storage and retrieval of large volumes of data. 0525162632
     </p>
     <h2>
      <span class="mw-headline" id="Simulation_of_WSNs">
       Simulation of WSNs
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Wireless_sensor_network&amp;action=edit&amp;section=23" title="Edit section: Simulation of WSNs">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h2>
     <p>
      At present, agent-based modeling and simulation is the only paradigm which allows the simulation of complex behavior in the environments of wireless sensors (such as flocking).
      <sup class="reference" id="cite_ref-Paper_17-0">
       <a href="#cite_note-Paper-17">
        [17]
       </a>
      </sup>
      Agent-based simulation of wireless sensor and ad hoc networks is a relatively new paradigm.
      <a href="/wiki/Agent-based_model" title="Agent-based model">
       Agent-based modelling
      </a>
      was originally based on social simulation.
     </p>
     <p>
      Network simulators like
      <a href="/wiki/OPNET" title="OPNET">
       OPNET
      </a>
      ,
      <a href="/wiki/NetSim" title="NetSim">
       NetSim
      </a>
      and
      <a class="mw-redirect" href="/wiki/NS2" title="NS2">
       NS2
      </a>
      can be used to simulate a wireless sensor network.
     </p>

      <span class="mw-headline" id="Other_concepts">

      <span class="mw-editsection">
       <span class="mw-editsection-bracket">

       <span class="mw-editsection-bracket">
        ]
       </span>

     </h2>
     <h3>
      <span class="mw-headline" id="Distributed_sensor_network">
       Distributed sensor network
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Wireless_sensor_network&amp;action=edit&amp;section=25" title="Edit section: Distributed sensor network">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      If a centralised architecture is used in a sensor network and the central node fails, then the entire network will collapse, however the reliability of the sensor network can be increased by using a distributed control architecture. Distributed control is used in WSNs for the following reasons:
     </p>
     <ol>
      <li>
       Sensor nodes are prone to failure,
      </li>
      <li>
       For better collection of data,
      </li>
      <li>
       To provide nodes with backup in case of failure of the central node.
      </li>
     </ol>
     <p>
      There is also no centralised body to allocate the resources and they have to be self organized.
     </p>
     <h3>
      <span class="mw-headline" id="Data_integration_and_sensor_web">
       Data integration and sensor web
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Wireless_sensor_network&amp;action=edit&amp;section=26" title="Edit section: Data integration and sensor web">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      The data gathered from wireless sensor networks is usually saved in the form of numerical data in a central base station. Additionally, the
      <a href="/wiki/Open_Geospatial_Consortium" title="Open Geospatial Consortium">
       Open Geospatial Consortium
      </a>
      (OGC) is specifying standards for interoperability interfaces and metadata encodings that enable real time integration of heterogeneous sensor webs into the Internet, allowing any individual to monitor or control wireless sensor networks through a web browser.
     </p>
     <h3>
      <span class="mw-headline" id="In-network_processing">
       In-network processing
      </span>
      <span class="mw-editsection">
       <span class="mw-editsection-bracket">
        [
       </span>
       <a href="/w/index.php?title=Wireless_sensor_network&amp;action=edit&amp;section=27" title="Edit section: In-network processing">
        edit
       </a>
       <span class="mw-editsection-bracket">
        ]
       </span>
      </span>
     </h3>
     <p>
      To reduce communication costs some algorithms remove or reduce nodes' redundant sensor information and avoid forwarding data that is of no use. As nodes can inspect the data they forward, they can measure averages or directionality for example of readings from other nodes. For example, in sensing and monitoring applications, it is generally the case that neighboring sensor nodes monitoring an environmental feature typically register similar values. This kind of data redundancy due to the spatial correlation between sensor observations inspires techniques for in-network data aggregation and mining
     </p>
